AI Interview Coach

01 — The Problem

Most people who struggle in job interviews don't lack knowledge. They lack practice under realistic conditions. A question you can answer calmly at your desk feels different when a stranger is watching the clock.

The options available today are expensive (coaching: ₹2,000–₹10,000 per session), logistically difficult (peer mock interviews require a willing, domain-knowledgeable friend), or too passive (watching YouTube tips doesn't build any kind of muscle memory).

"The problem isn't information scarcity — it's the absence of a safe, repeatable space to fail in before it counts."

This is the gap the AI Interview Coach targets: accessible, on-demand, judgment-free practice with feedback comparable to a coached session — at zero cost. The target user is a final-year student or early-career professional preparing for their first or second serious role.

02 — How It Works

1

Role & context setup

User picks a job role (e.g. "Product Manager — early-stage startup") and experience level. This seeds the system prompt that shapes the interviewer persona, question difficulty, and evaluation criteria — so the session is calibrated to the user's target, not generic.

2

Live interview simulation

Gemini API acts as the interviewer. Each turn generates a context-aware follow-up based on the user's previous answer — maintaining a conversational thread rather than pulling from a random queue. The model is instructed to probe gaps, not change topics arbitrarily.

3

Structured answer evaluation

After each answer, the model scores it across three dimensions: Clarity (was it easy to follow?), Specificity (were claims backed by detail?), and Relevance (did it actually answer the question?). Scores are returned as structured data — not a paragraph — so the UI renders them consistently every time.

4

Session synthesis

After 5–8 questions, the model generates a wrap-up: patterns it noticed across answers, the strongest moment, and 2–3 concrete things to practice before the real interview. This is the output users reported finding most valuable in informal testing.

03 — UX Decisions & Why

  • Decision 01

    No typing indicator / "AI is thinking" animation

    I removed the standard loading spinner. In a real interview, the other person doesn't signal they're processing your answer — there's just a pause. Removing the loading state made the experience feel less like a chatbot and more like a conversation. Three informal testers unprompted said it "felt more real."

  • Decision 02

    Enforced structured output via system prompt — not post-processed

    Early versions returned free-form feedback paragraphs that varied wildly in length and tone. I added a strict output schema directly in the system prompt (format: clarity:N | specificity:N | note:…). JavaScript parses this predictably. If the model breaks format (it occasionally does), there's a fallback to raw text — but I documented that failure state rather than pretending it doesn't exist.

  • Decision 03

    Fully generated questions — no question bank

    I chose generative questions over a curated database. This means every session genuinely differs and avoids the "I've seen this one before" problem that kills the training value of any repeat use. The tradeoff is occasional off-topic or poorly-calibrated questions. I document this openly rather than hiding it.

  • Decision 04

    Context window truncation after 3 exchanges

    Gemini processes the full conversation history per API call. After 5 turns, latency and cost climb noticeably. I truncated the live context to the last 3 exchanges while keeping the original system prompt intact — preserving role calibration and question tone without sending the full session on every call. This was an explicit product decision with a real tradeoff that I had to name and document.

04 — How AI Is Used (Specifically)

Dynamic system prompt construction

The model doesn't receive a static template. The system prompt is built at runtime from the user's setup choices: role, seniority, and interview type. This means the "PM at a startup" and "SDE at a large company" sessions have meaningfully different contexts — not just a different label on the same prompt.

Output structure enforcement

Rather than "give feedback on this answer," the prompt instructs the model to return structured output: clarity: N, specificity: N, note: "…". JS parses this directly into UI components. This gives the interface predictability and keeps the feedback scannable — not a wall of AI text.

Context-aware follow-up question logic

The system prompt includes: "Generate the next question by probing a gap or undeveloped thread in the candidate's last answer." If a user says "I used Agile" without elaborating, the model asks specifically about sprint planning or retrospectives — rather than moving to an unrelated topic. This makes sessions feel like real interviews, not surveys.

What AI does NOT touch

UI rendering, session state, navigation, and error handling are all plain JavaScript. AI handles three things only: question generation, answer evaluation, and session synthesis. This was a deliberate scope decision — keeping the AI layer thin, fast, auditable, and replaceable if a better model comes along.

05 — Limitations (Honest)

No voice input

Typing an answer is fundamentally different from speaking one. The product sidesteps the core anxiety of verbal delivery — which is the thing most people actually need to practice.

AI hallucinations in feedback

The model occasionally gives feedback that contradicts what the user actually said. There's no ground-truth checking layer. Users need to know this, not be surprised by it.

No domain-specific calibration

"PM at fintech" and "PM at gaming" should produce different interview depth. The current system doesn't handle industry specialisation well.

No session memory

Users can't track improvement over time. There's no comparison between session 1 and session 5. Without that, the coaching value plateau is invisible.

Next: Web Speech API

Browser-native speech-to-text for voice answers. No third-party SDK required — available in all modern browsers.

Next: Confidence indicators

Show which parts of feedback the model inferred vs. read directly — so users trust the feedback because they understand it, not despite not understanding it.

Next: Role-specific rubrics

Pre-built evaluation criteria for 5 common roles (PM, SDE, Designer, Data Analyst, Consultant) to improve feedback precision where it matters most.

Next: localStorage sessions

Save summaries locally so users can compare their own sessions week over week without requiring a backend or login.

06 — What I Learned

The hardest design problem wasn't the interface — it was managing user expectations of the AI. When a model gives you feedback on your answer, the instinct is to trust it completely. But LLMs can be confidently wrong. The real design challenge is communicating "this is useful signal, not ground truth" without undermining the product's entire value proposition.

I also learned that prompt engineering is a design practice, not a developer task. The structure, tone, constraints, and examples you put in a system prompt directly shape what users experience — it deserves the same iteration and documentation that a design component does.

"If I redid this project, I'd run 5 user interviews before writing a line of code. I built because I thought I understood the problem. I mostly did — but I made assumptions that cost me time I could have spent earlier."

01 UX Redesign

Zomato Refund & Dispute Resolution

When a food order goes wrong, Zomato's resolution flow puts up walls. Users who have legitimate complaints often give up — not because the problem is unsolvable, but because the UX makes the correct path harder than the path of least resistance (accepting a partial refund and moving on). That's not a bug. It's a design choice — and it erodes trust over time.

The core failure

The flow optimises for deflection, not resolution. A chatbot handles every case identically regardless of the actual issue, which is appropriate for maybe 30% of situations. The other 70% need a different path — one that doesn't currently exist.

My approach

Map every failure state (wrong item, missing item, cold food, late delivery, no delivery). Design a state-machine UI that matches the user's specific situation before they type a single word. Remove the chatbot from unambiguous, high-confidence scenarios entirely.

Where AI helps

Classify the issue type from the order data (delivery time, restaurant history, item mismatch patterns) before the user even opens the support flow. Pre-surface the likely issue: "Your order arrived 44 minutes late — was that a problem?" AI reduces steps, not just keystrokes.

What success looks like

Fewer users abandoning the resolution flow. Fewer manual support escalations for cases that are resolvable automatically. Users who feel the app is working for them — which is the only way to build the kind of trust that survives a bad delivery.

02 AI Feature

JobFill — Smart Job Application Autofill

The average active job seeker fills the same 20 fields across 50+ applications. ATS forms break standard browser autofill. The cognitive load of retyping identical information again and again doesn't just waste time — it actively discourages people from applying to roles they're qualified for.

The core failure

Current autofill works at the browser level: name, email, address. It doesn't understand resume content, can't interpret "describe a leadership experience in 200 words," and has no model of what the form is actually asking for. The gap between "what autofill can do" and "what job seekers need" is enormous.

My approach

A browser extension that parses the user's uploaded resume with an LLM, detects form field intent (not just field names — intent), fills fields with a visible confidence score per field, and highlights low-confidence fills for manual review before submission.

Key UX decision — trust design

Every AI-filled field shows a confidence indicator (high / medium / low). Low-confidence fills are highlighted — not silently filled and hoped for the best. One-click correction lets users teach the model what they actually want. Over time, the model improves for that specific user. The AI is legible — users trust it because they understand it, not despite not thinking about it.

What success looks like

Application time drops from ~20 minutes to ~5 for returning users. Application volume increases without increasing burnout. And — the harder metric — users report feeling like the tool is on their side, not making decisions for them.

✓ Can they explain decisions?

Yes — every project here documents not just what was built, but why one option was chosen over others, what was rejected, and what the tradeoffs were.

✓ Have they shipped anything?

Yes — two live products: the AI Interview Coach and SkillSwap. Both are accessible, not mockups. This matters more than a polished Figma deck that was never tested.

✓ Do they understand AI beyond surface level?

Yes — the case study details API architecture, prompt engineering decisions, context management, and failure state handling. This is practical AI knowledge, not trend adoption.

⚠ Do they have user research documentation?

Not yet in portfolio-quality format. Research was done but not documented in a shareable artefact. Next priority: affinity maps and persona documentation for the next project.

⚠ Is there visual design range?

The work is clean and functional, but the visual design vocabulary is limited to one aesthetic. Developing range across different visual styles is a stated goal for the next 6 months.

✓ Are they self-aware?

Yes — the portfolio explicitly names gaps and states how they're being addressed. Recruiters distrust candidates who claim no weaknesses. Self-awareness signals coachability.
